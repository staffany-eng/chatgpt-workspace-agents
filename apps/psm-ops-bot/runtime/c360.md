# Customer 360 Runtime

PSM Ops Bot uses Customer 360 as the customer context source of truth.

## Contract

- Base URL env var: `CUSTOMER360_BASE_URL`
- Token env var: `CUSTOMER360_INTERNAL_API_TOKEN`
- V1 access: all Customer 360 customers
- Auth: `X-Customer360-Internal-Token: $CUSTOMER360_INTERNAL_API_TOKEN`
- Do not rely on `Authorization` for C360 machine-to-machine calls; Cloud Run
  can treat that as platform auth before the app route sees it.

## Allowed Routes

- `GET /api/companies?q=<query>` for customer search.
- `GET /api/companies/{customerKey}/context?format=markdown` for compact account context.
- `POST /api/companies/{customerKey}/ask` for compiled customer-wiki Q&A.

All three MCP tools accept optional `slack_thread_url` for PS WEE flows. When present, verified C360 responses and blocked C360 failures can be copied to the central ops audit channel with the source Slack thread permalink.

Successful C360 tool results expose a canonical company link:

- `search_c360_customers` adds `c360_url` and `customer360_url` to each returned customer group, preferring C360-provided links and otherwise deriving `/companies/{customerKey}` from the resolved route key.
- `get_c360_account_context` and `ask_c360_customer_context` add top-level `c360_url` and `customer360_url` fields for the canonical customer key.
- Unresolved, blocked, and AA-channel redirect responses do not invent a Customer 360 link.

The bot must not use a personal `customer360_session` cookie.

## Data Boundary

Customer 360 serves compiled or compact context only. The bot must not read raw Slack, Intercom, WhatsApp, GCS source packs, or customer exports directly.

## Failure Behavior

If C360 returns 401/403, report `Confidence: blocked` with the internal auth issue. If C360 returns missing data, say what compiled source evidence is missing.

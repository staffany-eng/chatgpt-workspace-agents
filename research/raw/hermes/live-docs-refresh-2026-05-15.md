## Evidence Extracts

- The live docs expose both `llms.txt` and `llms-full.txt`; the May 15 refresh captured 98 curated index links and 153 full source pages.
- Compared with the May 10 refresh, the full bundle added six source pages and removed none, including worker lanes, Codex app server runtime, LSP, Line messaging, LLM plugin access, and video-generation provider plugins.
- Hermes positions itself as a self-improving, terminal-native agent with persistent memory, agent-created skills, and a messaging gateway spanning many platforms.
- Hermes setup spans Linux, macOS, WSL2, Termux, Nix, native Windows beta, Docker, SSH-like remote usage, and deployment profiles, so runtime assumptions should be host-specific.
- Hermes profiles are separate runtime homes with their own config, `.env`, `SOUL.md`, memories, sessions, skills, cron jobs, gateway state, and shareable non-secret distribution shape.
- Prompt assembly keeps identity, memory snapshots, user-profile snapshots, skills index, project context files, timestamp/session data, platform hints, and ephemeral overlays as separate layers.
- Context files use priority ordering and progressive discovery, with `SOUL.md` serving as global identity and lower-level project files loaded as needed instead of all at startup.
- Skills are procedural memory with progressive disclosure; skill packages can include references, templates, scripts, assets, environment prompts, config settings, platform filters, and conditional activation.
- Hermes can create or update skills through a self-improvement loop, and Curator exists to review generated skills for staleness, narrowness, duplication, and catalog quality.
- Built-in memory is bounded into `MEMORY.md` and `USER.md`, snapshotted at session start, and intentionally excludes raw dumps, session ephemera, context-file content, secrets, and other unsafe material.
- External memory providers such as Honcho are additive to built-in memory rather than replacements, giving semantic recall or user modeling while keeping file memory boundaries intact.
- Hermes tool architecture includes built-in tools, custom tools, MCP servers, ACP/API/tool gateway surfaces, per-server filtering, dynamic refresh, utility wrappers, and environment filtering for subprocesses.
- Messaging gateway docs cover a broad adapter set; Slack specifically uses Socket Mode, app-level and bot tokens, mention-first channel starts, thread continuation, app reinstall after scope changes, and optional channel-specific prompts or skill bindings.
- Cron supports one-shot and recurring automation, skill attachment, workdir binding, local/origin/platform delivery, quiet-success `[SILENT]` suppression, and script-only no-agent checks that avoid LLM usage.
- Kanban is separate durable work management backed by SQLite and gateway dispatch, while delegation is an in-turn fork/join tool for volatile parallel work.
- The new Kanban worker-lanes page makes queue execution more explicit: lanes bind named worker profiles, concurrency, and work classes rather than relying on one generic worker loop.
- Persistent goals keep a session objective alive across turns through judge-loop continuation and explicit pause, resume, and clear controls; they are distinct from cron and Kanban.
- Security guidance layers user authorization, dangerous-command approval, container/sandbox isolation, MCP credential filtering, context-file scanning, cross-session isolation, input sanitization, checkpoints, and rollback.
- Developer docs describe agent loop, prompt assembly, context compression/caching, gateway internals, session storage, provider runtime, tool runtime, and extension points for model, memory, image, video, LLM access, and context-engine plugins.
- Media and web features include browser supervision, web search, computer use, vision, image generation, text-to-speech, voice mode, Spotify, web dashboard, skins, RL-training, and trajectory export surfaces.
- Reference docs cover CLI, slash commands, profile commands, environment variables, tools, toolsets, MCP config, model catalog, skills catalog, optional skills catalog, and FAQ.
- For StaffAny bots, Hermes docs support the source-packet pattern: durable `SOUL.md`, config templates, skills, MCP contracts, Slack/runtime docs, health checks, and runbooks belong in repo control, while `.env`, sessions, memory, cron state, gateway state, logs, and unreviewed runtime learning stay in the live profile.
- For StaffAny bots, runtime learning should not be treated as product truth until promoted into a reviewed skill, reference, MCP contract, config template, test, or runbook.
- For StaffAny bots, adding capabilities should pick the right Hermes primitive: skill for repeatable procedure, MCP for external system boundary, cron for scheduled work, no-agent script for health checks, Kanban for durable task queues, delegation for in-turn parallel work, and memory only for confirmed recall.

## Evidence Extracts

- The docs site publishes machine-readable `llms.txt` and `llms-full.txt` entry points, and the May 10 full bundle covers 147 source pages across install, usage, features, messaging, integrations, guides, developer docs, and reference.
- Hermes profiles are separate home directories with their own `config.yaml`, `.env`, `SOUL.md`, memories, sessions, skills, cron jobs, and gateway state; profile sharing packages include non-secret app shape while credentials, memories, and sessions remain per machine.
- Hermes prompt assembly treats `SOUL.md`, frozen memory snapshots, user profile snapshots, skills index, context files, timestamp/session data, platform hints, and ephemeral overlays as distinct layers.
- Hermes context files use a priority order, load `SOUL.md` from `HERMES_HOME` as global identity, and progressively discover subdirectory context files during tool use to avoid startup prompt bloat.
- Hermes skills are procedural memory loaded with progressive disclosure; skill folders can contain references, templates, scripts, assets, environment prompts, config settings, platform filters, conditional activation rules, and external read-only skill directories.
- Hermes' self-improvement loop can create or update skills, while Curator exists to review stale, narrow, or duplicate agent-created skills and keep the skill catalog useful.
- Hermes memory is bounded into `MEMORY.md` and `USER.md`, snapshotted at session start, managed by memory tool actions, and intentionally skips raw dumps, session ephemera, context-file content, and secrets.
- Hermes external memory providers are additive to built-in memory; Honcho and other providers can add semantic search, user modeling, and cross-session context without replacing `MEMORY.md` or `USER.md`.
- Hermes MCP support covers local stdio and remote HTTP servers, startup discovery, prefixed tool registration, per-server include/exclude filtering, utility wrappers for resources/prompts, dynamic tool refresh, and stdio environment filtering.
- Hermes Slack support uses modern Socket Mode, requires bot and app-level tokens, starts conversations by mention in channels, continues active thread sessions, supports per-channel prompts and skill bindings, and requires reinstalling the Slack app after scope or event changes.
- Hermes cron jobs can run one-shot or recurring tasks, attach one or more skills, set a workdir, deliver to origin/local/platform targets, suppress successful quiet outputs with `[SILENT]`, and run script-only no-agent checks with zero LLM involvement.
- Hermes distinguishes volatile subagent delegation from durable Kanban work: `delegate_task` is an in-turn fork/join tool, while Kanban is a SQLite-backed queue with named profiles, comments, run history, and gateway-embedded dispatch.
- Hermes security docs define layered safeguards: user authorization, dangerous command approval, container isolation, MCP credential filtering, context-file scanning, cross-session isolation, input sanitization, and opt-in checkpoints/rollback for destructive file operations.
- Hermes persistent goals keep a session objective alive across turns with a judge loop and explicit pause/resume/clear controls; useful for long tasks, but separate from cron, Kanban, and normal prompt instructions.

## First-Pass Learning

Hermes' current docs reinforce the Agent Builder app-packet thesis:
keep durable app design in source-controlled files and skills, keep runtime state inside a named profile,
expose external systems through narrow tools or MCPs, and use cron/Kanban/delegation as different automation primitives
rather than one interchangeable "agent keeps working" bucket.
